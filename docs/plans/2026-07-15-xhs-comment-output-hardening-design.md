# 小红书评论区输出加固设计

## 背景

1.3.28 已能通过隐藏浏览器监听评论接口并采集超过 10 条主评论，但 2026-07-15 的三份真实笔记仍暴露出四类问题：DOM 兜底重复或噪音被追加为主评论、折叠回复归属丢失、点赞与时间未经规范化、诊断计数与最终 Markdown 不一致。页面内无签名直连接口持续返回 `root_unavailable`，真正有效的数据来自页面自身触发的已签名网络请求。

同批样本还出现了一段转写末尾的高置信错听结束语。它不属于评论链路，作为独立的小型清理规则处理，避免扩大成全文语义改写。

## 方案比较

### 方案 A：网络主数据 + 结构化 DOM 补充（采用）

- 页面滚动、展开按钮继续由真实页面执行，让小红书自己的脚本发出带签名请求；插件只监听响应。
- API 评论作为权威树。DOM 只有在能证明是主评论，或能唯一关联父评论时才允许补充。
- 无结构 DOM 在已有网络数据时只参与去重，不新增；完全没有网络数据时才作为降级主评论源。
- 最终 Markdown 生成后重新计数并写诊断。

优点是保留真实 DOM 补充能力，同时阻止重复与回复平铺。缺点是极少数无法识别结构的 DOM 独有评论会被保守丢弃。

### 方案 B：只用网络数据

最干净，但网络监听失败时评论区会完全缺失，无法接受。

### 方案 C：DOM 模糊匹配后全部追加

召回更高，但会继续产生错误归属和重复，不适合知识库成品。

## 详细设计

### 1. DOM 结构与合并规则

- DOM 收集项增加 `domRole`、`parentAuthor` 和可取得时的 `parentCommentId`。
- 回复节点优先按父评论 ID 归属；没有 ID 时仅在父作者唯一时归属。
- 已有网络主评论时，`domRole=unknown` 的新增项不进入最终树；无网络主评论时允许作为降级主评论。
- 诊断增加 `dropped`，记录被保守丢弃的无归属 DOM 项。

### 2. 指纹和噪音

- 指纹只用于比较，不修改展示正文；统一全半角标点、空白、尾部“展开”、小红书表情占位和非语义符号。
- 过滤空项、作者与正文相同、评论区统计/操作词，以及“问一问”自动摘要类文案。
- 网络 ID 仍是第一去重键；文本指纹作为跨 API/DOM 的补充键。

### 3. 页面采集

- 不把无签名页面 API 的 `root_unavailable` 当作主采集失败；真实页面监听是主通道。
- 滚动循环以“评论数与捕获响应数都稳定”为停止条件，允许更多轮次，但单轮等待缩短；达到 200 条或连续稳定即停止。
- 展开按钮继续触发折叠回复请求，响应按 root ID 合并，孤儿回复不平铺。

### 4. 输出与诊断

- Unix 秒/毫秒时间转为本地日期；已有“1天前·地区”文字保持原样。
- 点赞数字输出为“数字 赞”；原值只有“赞”时只输出一次。
- Markdown 生成后解析最终评论列表统计主评论和回复，使 `final_root/final_replies` 与文件肉眼可见内容一致。

### 5. 转写末尾清理

- 保留现有重复句、字幕署名清理。
- 只增加“错听结束语”高置信模式，且仅限最后一行；不调用生成模型、不改写正文中间内容。
- 清理后仍走现有空文本和质量异常检查。

## 测试

- 单元回归覆盖真实样本中的表情/展开差异、重复根评论、作者自重复、“问一问”、无归属回复、DOM 结构归属。
- 覆盖时间与点赞输出，以及 Markdown 最终计数。
- 覆盖错听结束语只在末尾删除、正文中出现时保留。
- 运行插件主测试、市场包测试、语法检查和 `git diff --check`。
