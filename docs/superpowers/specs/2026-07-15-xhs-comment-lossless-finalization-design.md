# 小红书评论无损收尾设计

## 背景与证据

正式插件 `1.3.32` 已能捕获小红书真实签名评论接口，但 2026-07-15 的三篇实测笔记仍显示两个残留问题：网络阶段捕获的回复数高于最终 Markdown 回复数；其中一个样本只捕获 1 页主评论，却捕获 11 页子回复后以 `network_idle` 停止。说明问题已从“拿不到接口”转为“滚动目标偏离主列表”和“评论树在后续合并中没有无损保证”。

## 目标与范围

仅修改 Obsidian 插件发布源、插件回归测试和发布文档。主评论上限保持 200，每条主评论回复上限保持 100。不修改小程序、云函数、登录方式、媒体下载、OCR、ASR 或用户数据。

## 方案

1. 浏览器网络评论树是权威主数据。后续 API 重放、Debugger 通用解析、DOM 和静态 HTML 只能补充字段或新增缺失项，不得让已有根评论或回复消失。
2. 合并每个阶段后计算评论 ID 集合。若候选结果缺少上一阶段的网络评论，通过 ID/父评论 ID 把缺失节点恢复到原父评论，并在诊断中记录恢复数。
3. 页面滚动容器优先选择包含主评论节点的评论列表及其可滚动祖先；带 `reply` / `sub-comment` 标记的嵌套容器不能获得主列表加分，避免滚动停在单个回复区。
4. 采集顺序先推进主评论列表，再逐轮点击“展开/更多回复”。回复响应先暂存，等主评论响应收齐后统一归属。无法归属的回复保留到最后一次匹配，不在首轮直接丢弃。
5. 停止条件同时参考主评论页是否 `has_more`、主评论页数量、根评论数量和容器是否移动；只要仍观察到新的主评论游标，就不能仅因 DOM 空闲退出。
6. Emoji/纯符号昵称参与跨来源去重时使用原昵称的 NFKC 结果作为后备键，避免同一 DOM 评论重复追加。

## 诊断与验收

诊断新增网络合并后的根/回复数量、恢复数量和最终损失数量。最终 Markdown 的根评论/回复数必须与最终评论树一致；`lost_root` 和 `lost_replies` 在正常保存中必须为 0。

回归覆盖：74 个根评论与 19 条回复经过多来源合并及二次保存仍为 74/19；回复响应早于主评论时最终正确归属；嵌套回复滚动容器不会胜过主评论列表；Emoji/符号昵称不会重复；200 条主评论边界不回退。

发布目标为 `1.3.33`。完成自动测试后安装到当前知识库，并按既有市场发布检查验证默认分支、标签、Release 资产、raw manifest 和本地 ZIP。
