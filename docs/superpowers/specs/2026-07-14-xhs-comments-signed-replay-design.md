# 小红书评论真实请求回放设计

## 目标

修复小红书评论区“主评论能抓到、折叠回复抓不到”的问题，继续支持最多 200 条主评论，并把网页真实加载的折叠回复挂回对应主评论。

## 已确认根因

现有页面内脚本直接拼接 `/api/sns/web/v2/comment/page` 和 `/comment/sub/page` 请求。三篇真实笔记的诊断均为 `pages=0; stop=root_unavailable; replies=0`，说明首个手工请求被平台拒绝。现有 Electron 调试器虽然能看到页面真实评论请求，但只把响应中的评论扁平化，无法区分主评论响应与子回复响应，也无法按 `root_comment_id` 建立父子关系。

## 方案

保留手工分页作为快速尝试；同时在隐藏、已登录的小红书页面中：

1. 监听 `Network.responseReceived` / `Network.loadingFinished`，保存真实评论接口的 URL 与响应体。
2. 按 URL 区分 `comment/page` 主评论和 `comment/sub/page` 折叠回复，从 URL 查询参数读取 `root_comment_id`。
3. 用现有评论规范化和分页合并逻辑，把子回复响应合并到对应主评论；没有父 ID 时只作为安全的平铺兜底，不伪造层级。
4. 扩展页面内“展开更多回复/查看回复”触发器和评论区域滚动，让网页自己发出带动态签名的真实请求。
5. 诊断优先记录真实浏览器网络来源、主评论数、回复数、请求数和停止原因，区分手工 API 失败与真实请求成功。

## 不变项

- 不绕过登录、验证码或平台权限；只复用用户已登录页面发出的请求和响应。
- 不改变评论 ID 优先去重、200 条主评论上限和 100 条回复上限。
- 不修改小程序、云函数、媒体下载、OCR 或转写链路。

## 验证

新增纯函数回归测试：给定一个主评论分页响应和一个带 `root_comment_id` 的子回复响应，结果必须生成带嵌套 `replies` 的主评论；同时检查渲染器已接入真实响应归类、回复触发器和安全诊断字段。
