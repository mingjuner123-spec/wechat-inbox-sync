# 小红书评论区完整采集设计

## 目标

在已登录的小红书会话中，默认采集最多 200 条主评论，并完整采集这些主评论可取得的折叠回复；不再把评论区仅限于首屏、已渲染 DOM 或一次网络响应。

## 范围

仅修改 Obsidian 插件发布源 `obsidian-plugin/wechat-inbox-sync/` 及其 Node 回归测试。不会改动小程序、云函数、用户数据或登录凭证存储方式。

## 采集路径

1. 使用现有持久化小红书 Electron session 打开笔记并捕获评论 API 的真实请求与响应。
2. 从第一个可用请求识别评论分页模板，在页面上下文中以 `credentials: include` 继续请求主评论页，读取 `cursor` 与 `has_more`，直到没有下一页、达到 200 条或平台拒绝请求。
3. 对声明存在更多回复的主评论，调用同会话的回复分页接口，读取所有可取得回复页。
4. 继续保留已存在的网络响应正文采集、受限的“展开回复/更多评论”点击和评论容器滚动，作为 API 字段变化、页面延迟或接口失败时的兜底。

## 数据模型与输出

- 评论使用平台 comment ID 去重；仅当 ID 不存在时，使用作者、内容、时间组合键。
- 回复保留在父评论的 `replies` 中，不展开成无关联的顶层评论。
- Markdown 继续采用嵌套列表输出。
- 当采集因 200 条安全上限、登录失效、接口拒绝或分页字段缺失停止时，写入不影响正文展示的 HTML 诊断注释，记录来源、主评论数、回复数、页数和停止原因；不记录 Cookie、授权头或原始请求体。

## 失败与降级

- 未登录、验证码或平台拒绝时，不伪造“已抓完”；保存正文和已获得评论，并记录原因。
- API 请求失败后继续使用 DOM 与网络响应中已经获得的评论。
- 不尝试绕过登录、验证码或平台访问限制。

## 验收与测试

- 四页主评论分页可聚合到 200 条。
- 有隐藏回复的评论会继续拉取回复页，且 Markdown 仍保留父子层级。
- 同作者同文案但不同评论 ID 不会被误删。
- `has_more=true` 但 cursor 缺失、会话拒绝和无登录态时，结果有明确停止原因且不会抛弃已采集评论。
