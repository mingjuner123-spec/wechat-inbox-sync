# 小红书评论完整采集与去重设计

## 目标

修复正式插件 `1.3.26` 在真实小红书笔记上的四个关联问题：主评论停在首批 10 条、部分折叠回复无法归属、API/DOM 评论重复、最终 Markdown 丢失回复缩进。主评论默认最多保存 200 条，每条主评论最多保存 100 条可取得回复。

## 范围

仅修改 Obsidian 插件发布源 `obsidian-plugin/wechat-inbox-sync/`、插件 Node 回归测试和发布文档。不修改小程序、云函数、登录凭证、媒体下载、OCR、ASR 或用户数据。

## 数据源优先级

1. 浏览器真实签名的 `comment/page` 和 `comment/sub/page` 响应是权威数据源，保留评论 ID、父评论 ID、游标和层级。
2. 页面内手工 API 请求仅作为诊断和快速尝试；失败时记录安全错误类别，不覆盖真实浏览器链路的状态。
3. DOM 与静态 HTML 只补充网络结果中不存在的评论，不再作为整批结果直接追加。
4. DOM 没有平台 ID 时，以标准化后的作者和正文匹配网络评论；展示时间不参与跨来源去重。

## 真实请求驱动

隐藏 Electron 页面加载后继续监听 DevTools Network 响应。页面脚本定位实际承载评论列表的可滚动容器，而不是固定滚动 `window`；每轮把该容器推进到底部并触发 `scroll`/`wheel`，等待新的评论响应或 DOM 数量变化。连续三轮没有新游标或新评论时停止，达到 200 条主评论或平台返回无更多数据时也停止。

对“展开、查看、更多回复”控件逐轮点击，让小红书页面自身发出带动态签名的 `comment/sub/page` 请求。所有回复响应先按 `root_comment_id` 暂存，待主评论响应收齐后统一挂载；找不到主评论的响应计入 `unmatchedReplies`，不伪造成顶层主评论。

## 统一评论模型与合并

内部评论保持 `id`、`rootId`、`parentId`、`author`、`content`、`time`、`likes`、`replies`。网络 ID 是唯一主键。DOM/静态数据只在作者与标准化正文均未命中任何网络主评论或回复时，作为无 ID 的补充主评论加入。

同一主评论来自多页或多次响应时执行字段合并：保留较完整正文、已有 ID 和全部去重回复，不能因先看到一个不含回复的副本而丢失后续回复。

## 输出与诊断

评论树合并完成后只生成一次 Markdown。子回复使用两个空格缩进的嵌套列表；通用 Markdown 清理对小红书评论启用保留列表缩进选项。

诊断注释同时记录网络原始数量和最终输出数量，包括根评论页数、回复页数、网络根评论、网络回复、最终根评论、最终回复、DOM 补充、去重数量、未归属回复和停止原因。诊断不写入 Cookie、Token、请求头、请求体或评论正文。

## 错误处理

- 登录失效、验证码或平台拒绝时保留已经取得的数据，并记录停止原因。
- 页面内手工 API 请求记录 HTTP/平台错误类别，不再空 `catch` 后伪装成正常结束。
- 评论容器不存在时退回 `window` 滚动，但诊断标记 `window_fallback`。
- 网络响应存在时，DOM 只补缺，不允许用 DOM 平铺结果覆盖网络层级。

## 验收标准

- 首屏 10 条且 `has_more=true` 的样本会继续捕获下一页，最多 200 条主评论。
- 子回复在响应早于或晚于主评论时都能挂回正确父评论。
- API 有 ID/毫秒时间、DOM 无 ID/相对时间的同一评论只输出一次。
- 最终诊断的 `finalReplies` 与 Markdown 中的回复行数量一致。
- Markdown 经最终存储清理后仍保留 `  - ↳` 缩进。
- 真实样本中不再出现三份测试笔记当前的大批重复项；不足 200 条时明确区分“确实没有更多”和“采集停止”。
