# 小红书匿名提取降级修复设计

## 目标

保留小红书“不登录优先提取”的现有体验。当匿名请求只返回“小红书 - 你的生活兴趣社区”等通用落地页时，插件改用持久化的小红书浏览器会话重新提取；仍无法获得真实笔记时明确报为可重试失败，不再把分享文案、默认封面或空评论误判为完整内容。

## 已确认根因

- 正常样本通过匿名或浏览器签名响应拿到了真实标题、正文、图片和评论，诊断为 `source=browser-network`。
- 失败样本只拿到通用落地页，诊断为 `source=page-api`、`root=0`、`root_unavailable`、`window_fallback`。
- 1.3.46 的隐藏浏览器已经返回 `renderedXiaohongshuPage.html`，但同步流程只消费其中的评论，没有用渲染 HTML 重新提取正文和图片。
- `hasReadableXiaohongshuGraphicContent` 只要看到任意图片，或兜底分享文案超过 20 个字符，就会判定提取成功，因此通用落地页被写入并标记为已同步。

## 方案比较

### 方案 A：只收紧成功判断

拒绝通用标题、默认图片和分享文案兜底。改动最小，但原本可由隐藏浏览器恢复的内容会直接失败，降低成功率。

### 方案 B：匿名优先 + 浏览器渲染回退（采用）

匿名 HTML 有真实内容时保持现有快速路径；识别为通用落地页时，复用现有持久浏览器会话，将渲染 HTML 再次交给同一提取器，并选择更完整的结果。渲染后仍无真实内容才返回可重试错误和登录提示。

### 方案 C：新增服务端抓取或绕过平台风控

可能提高部分链接成功率，但会引入服务端成本、Cookie/账号安全和平台合规风险，本次不采用。

## 数据流

1. 解析短链，匿名请求最终 URL。
2. 从匿名 HTML 提取标题、正文、图片和视频。
3. 判断结果是否为真实笔记：通用标题、默认图片、纯分享文案不能单独构成成功。
4. 若匿名结果真实，直接继续评论和 OCR 流程，不要求登录。
5. 若匿名结果是通用落地页，调用现有 `renderXiaohongshuPageWithElectron`。
6. 从其 `html` 重新提取笔记正文和图片，同时保留同一次渲染得到的评论诊断。
7. 比较匿名结果和渲染结果，优先使用真实标题、较长正文、真实图片较多的结果。
8. 两条路径都失败时抛出明确的可重试错误；同步记录不标记为已完成，并提示用户在插件专用窗口登录小红书后重试。

## 错误与安全边界

- 不自动登录，不读取 Chrome Cookie，不绕过验证码或平台风控。
- 没登录不等于立即失败；只有匿名和浏览器渲染都无法取得真实笔记时才提示登录。
- 评论 `root_unavailable` 本身不代表正文失败；只有它与通用落地页、无真实正文/图片同时出现时才判定整篇不可用。
- 已经误标为同步完成的历史记录需要人工或后台重置为 `pending`，本次插件修复不自动修改线上历史数据。

## 测试与验收

- 匿名 HTML 已完整：不调用浏览器正文回退，结果保持完整。
- 匿名 HTML 通用、渲染 HTML 完整：使用渲染后的真实标题、正文和图片。
- 匿名与渲染都通用：当前记录同步失败且不调用 `/synced`。
- 通用落地页中的默认图片和长分享文案不再通过“可读图文”判断。
- 已有小红书评论合并、图文 OCR、视频跳过 OCR 的回归测试继续通过。
