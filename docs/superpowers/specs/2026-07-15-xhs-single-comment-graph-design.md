# 小红书单一评论树重构设计

## 背景与根因

`1.3.27` 到 `1.3.33` 先后补上了真实签名响应捕获、评论容器滚动、延迟回复归属、跨来源去重和无损恢复，但 2026-07-15 的三篇真实笔记仍显示：网络阶段合计捕获 64 条回复，最终 Markdown 仅保留 30 条，且出现 `[doge]` 与无 emoji、绝对时间与相对时间的同一评论重复。

这些现象不是单一去重规则遗漏，而是数据生命周期没有收束：渲染浏览器阶段已经把网络、DOM 和延迟回复合成一次评论树，回到同步主流程后又把该树与静态 HTML 再次合并并重新标准化。第二次合并既失去了“网络节点不可减”的主数据边界，也让无 ID 的 DOM/静态副本再次获得新增根评论的机会。采集循环又把回复增长计入整体进展，回复持续到达时会掩盖主评论列表已经停止推进。

## 目标

1. 一条小红书记录在整个同步流程中只拥有一个最终评论图，且只生成一次 Markdown。
2. 网络评论 ID 是权威节点身份；同 ID 根评论和回复在任何后续阶段都不得消失。
3. 无 ID fallback 只补字段或补真正缺失的评论，不与网络树平级竞争。
4. 主评论分页进展与回复展开进展分别计数，回复增长不能伪装成主评论分页仍在推进。
5. 正常完成时满足根评论与回复守恒；无法证明完整时明确标记为 partial，而不是静默输出“完整”结果。

## 非目标

- 不修改小程序、云函数、登录、支付、Pro、媒体下载、ASR 或 OCR。
- 不尝试伪造小红书动态签名，不用无签名的页面内 `fetch` 作为完整性依据。
- 不承诺平台未返回或登录态不可见的评论一定可取得。
- 本轮先安装本地候选版验证，不在真实样本通过前发布插件市场。

## 单一评论图

评论图以根评论 ID 和回复 ID 为两个独立索引。每个节点保留 `id`、`rootId`、`parentId`、作者、正文、时间、点赞和回复列表。处理顺序固定为：

1. 把浏览器真实网络响应中的根评论和回复写入图。
2. 回复先于根评论到达时，按 `rootId` 放入 pending 集合；根评论出现后挂回。
3. DOM 与静态 HTML 只通过补充入口进入：优先按 ID 命中，其次按“作者 + 完整标准化正文”命中；命中时只填空字段，不重建节点。
4. 仅当 fallback 没有 ID 且无法命中任何已有根评论或回复时，才可新增无 ID 根评论。
5. 最终按根评论首次出现顺序和回复首次出现顺序渲染一次 Markdown。

跨来源正文标准化使用 NFKC、空白折叠，并移除纯展示型方括号 emoji 标记（例如 `[doge]`、`[笑哭R]`）。时间、点赞数不参与无 ID 身份。两个不同的网络 ID 即使正文相同也必须保留，避免误删真实复读评论。

## 单次收尾边界

隐藏浏览器返回的 `comments` 即为最终评论图。同步主流程不得再把它与初始静态 HTML 做第二次通用合并；只有隐藏浏览器完全没有生成评论图时，才使用静态 HTML 作为降级结果。最终 Markdown、统计和诊断必须由同一个 `finalizeXiaohongshuComments` 入口同时产生，避免“树是一份、Markdown 是另一份”。

## 采集进展与完整性

采集循环分别记录：

- 唯一根评论数与根评论网络响应数；
- 唯一回复数与子回复网络响应数；
- 主评论容器是否移动、游标是否变化；
- pending/unmatched 回复数。

主评论阶段的 `progressed` 只允许由根评论、根评论请求或主容器位置/高度变化触发，回复数量增长不能重置主评论空闲计数。回复展开可继续运行独立的收尾轮次。若平台仍声明 `has_more`，但根评论连续空闲达到上限，诊断写 `partial=1` 和 `stop=root_idle_with_more`。

## 守恒与诊断

最终入口计算网络阶段与最终图的 ID 集合：

- `lost_root = captured_root_ids - final_root_ids`
- `lost_replies = captured_reply_ids - final_reply_ids`

正常完成必须为 0。诊断还记录 `captured_root`、`captured_replies`、`final_root`、`final_replies`、`fallback_added`、`deduped_fallback`、`pending_replies`、`partial` 和停止原因。诊断不写 Cookie、签名、Token、请求正文或评论正文。

## 测试与验收

先用脱敏的生产形态 fixture 做端到端红灯：

- 74 根/19 回复经过浏览器阶段和同步收尾后仍为 74/19；
- `[doge]`/无 emoji、绝对/相对时间副本只输出一次；
- 回复先到根后到仍归属正确；
- 只有回复增长时，主评论进展判定为 false；
- 不同 ID 的相同文案仍保留两条；
- 最终 Markdown 统计与评论图统计完全一致。

本地候选版安装到当前知识库后，重新同步现有真实样本。只有在诊断显示 `lost_root=0`、`lost_replies=0`，可见评论没有跨来源重复，并且不完整场景被明确标为 partial 时，才进入市场发布流程。
